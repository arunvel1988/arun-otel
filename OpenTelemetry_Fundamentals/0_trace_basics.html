Tracing Fundamentals

Understanding Trace, Span, Trace ID, Span ID, Parent Span, Child Span, and Distributed Tracing using a real E-Commerce example.

Real E-Commerce Request Flow

Imagine a customer placing an order in an e-commerce application.

Frontend
→
Checkout Service
→
Payment Service
→
Inventory Service
→
Notification Service
Customer clicks "Place Order" Frontend ↓ Checkout Service ↓ Payment Service ↓ Inventory Service ↓ Notification Service

What is a Trace?

A Trace represents the complete journey of a request across multiple services and systems.

  • Tracks a request end-to-end
  • Helps identify bottlenecks
  • Shows service dependencies
  • Used in distributed systems
Trace Example: Customer places order ↓ Checkout API called ↓ Payment API called ↓ Inventory updated ↓ Email notification sent Entire flow = ONE TRACE

What is a Span?

A Span represents a single operation inside a Trace.

Checkout Span

Tracks checkout API execution.

Payment Span

Tracks payment processing operation.

Inventory Span

Tracks stock update operation.

Trace ├── Span: Checkout API ├── Span: Payment Service ├── Span: Inventory Service ├── Span: Notification Service

What is Trace ID?

Trace ID is a unique identifier assigned to the complete request journey.

  • Same Trace ID shared across all services
  • Helps correlate logs and spans
  • Used to track requests across systems
Trace ID: abc123xyz789 Frontend Span Checkout Span Payment Span Inventory Span All share SAME Trace ID

What is Span ID?

Span ID uniquely identifies an individual span inside a trace.

Service Span ID
Checkout Service span-101
Payment Service span-102
Inventory Service span-103
One Trace ↓ Multiple Spans ↓ Each Span has unique Span ID

Parent Span and Child Span

Spans can have parent-child relationships.

Parent Span: Checkout API ├── Child Span: Payment Service ├── Child Span: Inventory Service ├── Child Span: Notification Service
  • Main operation becomes Parent Span
  • Sub-operations become Child Spans
  • Helps visualize dependency flow

What is Distributed Tracing?

Distributed tracing tracks requests across multiple microservices.

Frontend
→
Checkout API
→
Payment API
→
Inventory API
Without Distributed Tracing: Very difficult to identify where failures happen. With Distributed Tracing: Engineers can identify: - Slow APIs - Failed services - Latency bottlenecks - Dependency failures

What is Context Propagation?

Context propagation means passing Trace ID and Span information across services.

Frontend sends request ↓ Trace ID attached to HTTP headers ↓ Checkout Service receives Trace ID ↓ Payment Service receives same Trace ID ↓ Entire request linked together
  • Keeps request connected across systems
  • Very important in microservices
  • Used heavily in Kubernetes environments

Complete Observability Flow

Customer Request ↓ Trace Created ↓ Multiple Spans Generated ↓ Trace ID Propagated ↓ Telemetry Exported ↓ Grafana / Tempo / Jaeger Visualization
