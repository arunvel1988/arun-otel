OpenTelemetry Logging API

Creating intelligent logs for distributed applications

Why Do We Need OpenTelemetry Logging?

Traditional applications already create logs.

Example:
Payment failed User id 500 Bank timeout

This is useful for humans. But modern microservices need more information.

Questions:
  • Which service generated this?
  • Which request created this error?
  • Which trace belongs to this log?
  • Which customer journey failed?

OpenTelemetry makes logs connected and meaningful.

Traditional Logging Problem

Checkout Service ERROR payment failed
↓
Payment Service ERROR timeout
↓
Database Service ERROR connection issue

All errors are separate. We cannot easily know they belong to the same customer request.

OpenTelemetry Logging Solution

OTel attaches context information automatically.

Checkout Log trace_id = abc123 Payment Log trace_id = abc123 Database Log trace_id = abc123

Now one trace id connects the complete journey.

What is an OpenTelemetry Log Record?

A Log Record is a structured event created by an application.

Instead of random text, OTel creates a standard object.

Log Record { Timestamp Severity Body Attributes Trace ID Span ID Resource Information }

Structured Log Record Example

Ecommerce checkout failure:
{ "time": "2026-01-01 10:00:00", "severity": "ERROR", "body": "Payment failed", "trace_id": "abc123", "span_id": "xyz999", "attributes":{ "user_id":"500", "order_id":"900", "payment_gateway":"stripe" } }

Every field has meaning. Machines can search and analyze this easily.

Important Parts of Log Record

Field Meaning
Timestamp When event happened
Severity Error level INFO, WARN, ERROR
Body Main log message
Attributes Extra information
Trace ID Connect logs with request journey
Span ID Connect log with exact operation

What Are Attributes?

Attributes are extra key-value details added to logs.

Example:
Message: Checkout completed Attributes: order_id=1001 amount=5000 country=India payment=success

Instead of searching text, we filter using attributes.

Resource Attributes

Resource attributes describe who created the log.

Example:
{ service.name="checkout-service" environment="production" region="asia" version="v2" }
Useful questions:
  • Show only checkout-service logs
  • Show production errors
  • Compare version v1 and v2

Context Propagation into Logs

This is the most powerful feature.

When a request moves between services, OpenTelemetry carries context.

Customer Checkout trace_id abc123
↓
Cart Service trace_id abc123
↓
Payment Service trace_id abc123
↓
Database trace_id abc123

All logs automatically contain the same trace id.

Without Context vs With Context

Without OTel Context

payment failed database slow checkout error

Cannot know relationship.

With OTel Context

trace_id=abc123 checkout started trace_id=abc123 payment failed trace_id=abc123 database timeout

One search shows complete failure journey.
