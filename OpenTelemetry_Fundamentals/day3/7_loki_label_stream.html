Loki Indexing Model

Understanding Labels, Streams and Why Loki Is Different

The Question Loki Tries To Solve

Imagine an ecommerce company generating millions of logs every day.

Payment failed Checkout started Checkout completed Product added to cart User logged in

How can Loki store and search all these logs efficiently?

The answer lies in Labels and Streams.

How Traditional Log Systems Work

Systems like Elasticsearch index every word.

ERROR payment failed for order 123

Indexed words:

ERROR payment failed order 123

Every word becomes searchable.

This is powerful but expensive.

How Loki Thinks Differently

Loki does NOT index every word.

Loki only indexes labels.

Labels: service="payment" environment="prod" level="ERROR" Message: Payment failed for order 123

Only the labels are indexed. The message itself is not indexed.

What Is A Label?

A label is metadata describing a log.

{ "service":"payment", "environment":"prod", "level":"ERROR" }

Labels answer questions like:

  • Which service generated this log?
  • Which environment?
  • What severity level?

Real Ecommerce Example

{ "service":"payment", "environment":"prod", "level":"ERROR", "message":"Bank timeout for order 123" }

Loki indexes:

service=payment environment=prod level=ERROR

Loki does NOT index:

Bank timeout for order 123

What Is A Stream?

This is the most important concept in Loki.

A Stream is a collection of logs having exactly the same labels.

Stream Example 1

Labels: service=payment environment=prod level=ERROR

All these logs belong to the same stream:

Bank timeout order 123 Bank timeout order 456 Card declined order 789 Payment gateway unavailable

Stream Example 2

Labels: service=payment environment=prod level=INFO

These logs belong to another stream:

Payment successful Payment successful Payment successful

Stream Example 3

Labels: service=checkout environment=prod level=INFO
Checkout started Checkout completed Checkout started

Different labels means different stream.

Visualizing Streams

Stream A service=payment level=ERROR
↓
Bank timeout Card declined Gateway unavailable

Stream B service=payment level=INFO
↓
Payment successful Payment successful

Stream C service=checkout level=INFO
↓
Checkout started Checkout completed

What Does Loki Actually Index?

Only labels.

service=payment environment=prod level=ERROR

Loki's index points to the location of the logs.

service=payment level=ERROR ↓ Chunk 123

How A Query Works

Suppose Grafana sends:

{service="payment"}

Loki does not search every log.

Instead it:

Check Index
↓
Find Matching Streams
↓
Read Relevant Chunks
↓
Return Logs

Why Loki Is Cheap

Elasticsearch Loki
Indexes every word Indexes labels only
Large index Small index
Higher storage cost Lower storage cost
Higher memory usage Lower memory usage

Bad Label Example

Never use labels that create millions of unique values.

user_id=1001 user_id=1002 user_id=1003 user_id=1004 ...

Every unique label combination creates a new stream.

This causes High Cardinality.

Why High Cardinality Is Dangerous

1 Million Users
↓
1 Million Different user_id Labels
↓
1 Million Streams
↓
Slow Loki High Memory Usage

Good Labels

service environment level region team version

These values do not change frequently.

They create a manageable number of streams.

Bad Labels

user_id session_id request_id order_id transaction_id

These values are usually unique.

Avoid using them as labels.

Memory Rule
