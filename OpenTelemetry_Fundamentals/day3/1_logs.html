Logging Essentials

Understanding logs in modern microservices

What Are Logs?

Logs are messages written by applications to explain what happened.

Example ecommerce application:

Customer opens website
↓
Adds product to cart
↓
Makes payment

During this journey application writes messages.

User opened homepage Product added to cart Payment successful Order created

These messages are logs.

Why Do We Need Logs?

Metrics tell us: Something happened.

payment_error_total = 100

But logs explain: Why it happened.

Payment failed Reason: Bank timeout User: 12345 Order: 98765

1. Unstructured Logs

Normal text messages written by applications.

Example:
Payment failed for user 1001 because bank timeout
Problems:
  • Difficult for machines to search
  • No fixed format
  • Hard to create dashboards
  • Difficult in large systems
Example problem:
Developer 1: payment failed user=100 Developer 2: ERROR payment issue customer 100 Same event Different format

2. Structured Logs

Structured logs store information in fixed fields.

Example:
{ "level":"ERROR", "service":"payment", "user_id":"1001", "message":"payment failed", "reason":"bank timeout" }
Now computers understand every field.

Structured vs Unstructured

Unstructured Structured
Payment failed user 100 "user_id":"100"
Only humans understand Machines understand
Hard searching Easy filtering
Bad for microservices Best for microservices

3. JSON Logging

JSON is the most common structured log format.

Why JSON?
  • Every language supports it
  • Easy to search
  • Works well with log systems
Example:
{ "time":"2026-01-01 10:00:00", "level":"INFO", "service":"checkout", "message":"checkout completed", "order_id":"500" }

4. Contextual Logging

In microservices one request travels through many applications.

Example:
Frontend
↓
Cart Service
↓
Payment Service
↓
Shipping Service
Question:

If payment fails, how do we find all logs belonging to one customer request?

Answer: Context.

Trace ID and Span ID

Trace ID

Unique ID for the complete request journey.

Example:
Customer checkout request trace_id = abc123
Every service keeps same trace id:
Cart Service trace_id=abc123 Payment Service trace_id=abc123 Shipping Service trace_id=abc123

Span ID

Unique ID for one operation inside the journey.

Example:
Trace ID: Full checkout journey Span IDs: Cart operation Payment operation Database operation

Contextual Log Example

{ "service":"payment", "level":"ERROR", "message":"bank timeout", "trace_id":"abc123", "span_id":"xyz789" }
Now engineers can find:
  • Exact failed request
  • Related logs
  • Related traces

Logging Best Practices for Microservices

Practice Reason
Use JSON logs Easy searching and filtering
Always add service name Know which application created log
Add trace id Connect logs with traces
Use proper levels Separate normal and error events
Avoid sensitive data Protect users

Log Levels

DEBUG: Developer details INFO: Normal application events WARN: Something unusual ERROR: Something failed

Final Memory Rule

Logs: Explain what happened Structured Logs: Organized information JSON: Common log format Trace ID: Connect complete request journey Span ID: One step inside journey Microservices: Always log with context
