Grafana Alloy for Logs

Collecting logs from applications and servers

Why Do We Need Alloy for Logs?

Applications generate logs continuously.

Example ecommerce application:
INFO user opened website INFO product added to cart ERROR payment failed INFO order created

But these logs are sitting inside different places.

  • Application files
  • Docker containers
  • Linux servers
  • Kubernetes pods

Alloy's job is to collect all logs and send them to Loki.

Complete Logging Flow

Flask Ecommerce Application Creates logs
↓
Logs stored somewhere File / Docker / Server
↓
Alloy Receiver Collect logs
↓
Loki Store logs
↓
Grafana Search logs

What is a Log Receiver?

Receiver means input.

It tells Alloy: Where should I collect logs from?

Receiver answers: Read this file Watch this container Read this server journal Accept logs from another tool

1. File Log Receiver

Used when applications write logs into files.

Example:
Flask App
↓
/var/log/app.log
↓
Alloy filelog receiver
Log file:
/var/log/ecommerce.log INFO checkout started ERROR payment failed
Alloy watches the file. Whenever new lines appear, Alloy collects them.

Filelog Configuration Example

otelcol.receiver.filelog "app" { include = [ "/var/log/ecommerce.log" ] output { logs = [ otelcol.exporter.otlp.default.input ] } }
Meaning:
  • Watch ecommerce.log
  • Read new log lines
  • Forward them

2. Docker Log Receiver

Most modern apps run inside containers.

Example:
Docker Container flask-app
↓
Container stdout logs
↓
Alloy Docker receiver
Container output:
docker logs flask-app INFO cart added ERROR checkout failed

Alloy collects container logs automatically.

Docker Logs Production Example

Container: checkout-service Logs: { service:"checkout", message:"payment failed", trace_id:"abc123" } Alloy collects ↓ Loki stores

3. Journald Receiver

Linux systems store service logs using systemd journal.

Example:
journalctl -u nginx Started nginx Request failed Service restarted
Flow:
Linux Service
↓
systemd journal
↓
Alloy journald receiver
Used for:
  • Linux servers
  • System services
  • Infrastructure logs

4. Fluent Receiver

Many companies already use Fluent Bit or Fluentd.

Instead of replacing them:

Alloy can receive logs from them.

Applications
↓
Fluent Bit
↓
Alloy Fluent Receiver
↓
Loki

Which Receiver Should I Use?

Scenario Receiver

Linux system services journald
Existing Fluent setup fluent receiver

Final Memory Rule

Application: Creates logs Receiver: Collects logs from source filelog: Reads files docker: Reads containers journald: Reads Linux logs fluent: Accepts Fluent logs Alloy: Moves logs to Loki
