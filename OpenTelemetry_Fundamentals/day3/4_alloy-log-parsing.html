Log Parsing with Alloy

Turning raw log lines into searchable information

Why Parsing Is Required?

Applications produce logs as plain text.

Example:
2026-06-15 10:00:00 ERROR Payment Failed User=123 Order=999

Humans can understand this. Computers struggle to understand individual pieces.

Questions:
  • Which user failed?
  • Which order failed?
  • How many ERROR logs?
  • Show only Order=999 logs

Parsing converts one text line into structured fields.

Before and After Parsing

Before Parsing

"2026-06-15 10:00:00 ERROR Payment Failed User=123 Order=999"

After Parsing

timestamp = 2026-06-15 10:00:00 level = ERROR message = Payment Failed user_id = 123 order_id = 999

Now Grafana and Loki can search these fields individually.

1. Regex Parsing

Regex means pattern matching.

Useful when logs are plain text.

Raw log:
ERROR Payment Failed User=123 Order=999
Regex:
(ERROR) (.*) User=(\d+) Order=(\d+)
Extracted fields:
level = ERROR message = Payment Failed user_id = 123 order_id = 999

Alloy can use regex processors to extract these fields automatically.

Ecommerce Regex Example

Raw Log: INFO Checkout Success User=500 Product=Football
After Regex Parsing:
level = INFO action = Checkout Success user_id = 500 product = Football

Now Grafana can search:

user_id=500
or
product=Football

Problem With Regex

Regex works but becomes difficult as log formats change.

Example:
INFO Payment Failed User=100 ERROR User=100 Payment Failed WARNING Payment Failure User=100

One regex may not work for all formats.

This is why modern applications prefer JSON logging.

2. JSON Parsing

JSON logs are already structured.

Application log:
{ "level":"ERROR", "user_id":"123", "order_id":"999", "message":"Payment Failed" }

Alloy simply reads the fields. No complex regex needed.

JSON Parsing Example

Application Log: { "service":"checkout", "user_id":"500", "amount":"2500", "message":"Checkout Complete" }
Parsed Result:
service = checkout user_id = 500 amount = 2500 message = Checkout Complete

This is why JSON logging is considered best practice.

Regex vs JSON

Regex JSON
Works on text logs Works on structured logs
Harder to maintain Easy to maintain
Complex patterns Direct field extraction
Legacy applications Modern applications

3. Severity Mapping

Different applications use different words for log levels.

Example:
ERR ERROR SEVERE CRITICAL FAIL

All mean something bad happened.

Severity Mapping converts different words into standard levels.

Severity Mapping Example

Before:
ERR Payment Failed SEVERE Database Down WARN Slow Query
After Mapping:
ERROR Payment Failed ERROR Database Down WARN Slow Query

Now dashboards become consistent.

Why Severity Mapping Matters?

Imagine three microservices.
Payment Service: ERR Checkout Service: ERROR Database Service: SEVERE

Without mapping:

Searching ERROR misses ERR and SEVERE

After mapping everything becomes searchable.
