Grafana Loki Architecture

How billions of logs are stored and searched efficiently

Why Loki Exists?

Modern applications generate huge amounts of logs.

Example ecommerce company:
Checkout Service 10000 logs/min Payment Service 20000 logs/min Cart Service 50000 logs/min

Storing and searching millions of logs becomes expensive. Loki solves this problem.

Traditional Logging Problem

Systems like Elasticsearch index every word.

Example:
ERROR payment failed for user 500 Indexed: ERROR payment failed user 500
Problems:
  • Huge storage usage
  • Expensive indexing
  • More infrastructure cost

Loki's Different Approach

Loki does not index complete log text.

Loki indexes only labels.

Example:
Labels: service="payment" environment="prod" level="error" Actual Log: Payment failed for user 500

Small index. Cheap storage. Fast filtering.

1. Distributor

Distributor is the front door of Loki.

Example:
Incoming: service=checkout message="payment started"
Distributor responsibilities:
  • Receives logs from Alloy
  • Validates log data
  • Checks tenant
  • Sends logs to ingesters
Simple thinking:
Distributor = Reception desk

2. Ingester

Ingester receives logs from Distributor.

Responsibilities:
  • Accept incoming logs
  • Group logs by labels
  • Create compressed chunks
  • Upload chunks to storage
Example:
Labels: service="payment" Logs: payment started payment completed payment failed Become one chunk

What is a Chunk?

A chunk is a compressed group of logs.

Example:
Chunk: Time: 10AM - 11AM Labels: service=cart Contains: Thousands of cart logs

Loki stores chunks instead of individual files.

Object Storage

Chunks are uploaded into cheap long term storage.

Ingester Creates Chunk
↓
S3 / MinIO Stores Chunk
Benefits:
  • Store months or years of logs
  • Cheap storage
  • Easy scaling

Index in Loki

Index tells Loki where logs are stored.

Example:
Index: service=payment time=10AM Location: chunk-file-123

Index is small because only labels are indexed.

Querier

Querier answers LogQL queries.

Example Grafana query:
{service="payment"}
Querier:
  • Checks index
  • Finds correct chunks
  • Reads logs
  • Returns result

Query Frontend

Large searches can be heavy. Query frontend improves performance.

Example:
Search: 30 days payment logs Query frontend splits: Day 1 Day 2 Day 3 Runs faster

Compactor

Over time many index files are created.

Compactor:
  • Merges index files
  • Applies retention rules
  • Deletes old logs
Example:
Keep logs: 90 days Delete: Older than 90 days
