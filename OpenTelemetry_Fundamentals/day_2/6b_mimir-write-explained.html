Mimir Write Architecture Explained

From customer action to long term metric storage

Step 1: Customer Does Something

Imagine an ecommerce application. A customer clicks Add To Cart.

Customer clicks Add To Cart button
↓
Flask API runs /cart endpoint
↓
Application increases metric cart_count_total + 1

Step 2: Metric is Created

Inside the application OpenTelemetry creates a metric.

Example: cart_count_total { service="cart-service" environment="production" } value = 500

This is only inside application memory currently. It has not reached Mimir yet.

Step 3: Alloy Collects the Metric

Grafana Alloy receives or collects this metric.

Application Metrics
↓
Grafana Alloy
Alloy can:
  • Add labels
  • Remove unwanted metrics
  • Prepare data for storage

Step 4: What is Remote Write?

Remote Write simply means:

Send metrics over HTTP from collector to metrics database
Example:
Alloy has: cart_count_total=500
↓ HTTP POST ↓
Mimir receives data

This HTTP request is called a Remote Write request.

Step 5: Mimir Distributor Receives Data

The first component inside Mimir is Distributor.

Remote Write arrives cart_count_total checkout_total payment_total
↓
Distributor
Distributor checks:
  • Is this metric valid?
  • Which customer/team owns it?
  • Where should this metric go?

Understanding Tenant

A tenant means a separate customer or team inside the same Mimir system.

Example company:
Same Mimir Cluster Tenant A: Payment Team Metrics Tenant B: Cart Team Metrics Tenant C: Search Team Metrics

Distributor identifies the tenant before storing data.

Step 6: Distributor Sends to Ingester

Distributor itself does not store metrics. It sends them to ingesters.

Distributor
↓
Ingester Stores incoming metrics
Example:
Incoming: cart_count_total 10:00 = 100 10:10 = 500 Ingester stores this time series

Step 7: Ingester Creates Blocks

Metrics are stored as time based blocks.

Example:
Block 1: 10 AM - 12 PM cart metrics payment metrics Block 2: 12 PM - 2 PM cart metrics payment metrics

Step 8: Blocks Go To Long Term Storage

Ingester creates blocks
↓
Object Storage S3 / MinIO

This is why Mimir can store years of metrics.

Complete Write Path

Customer Add Cart
↓
Flask creates metric cart_count_total
↓
Alloy collects and processes
↓
Remote Write request
↓
Mimir Distributor validate and route
↓
Ingester store recent metrics
↓
S3 / MinIO long term storage

Simple Memory Rule

Application: Creates metric Alloy: Collects metric and sends Remote Write: Delivery truck carrying metrics Distributor: Security gate + traffic controller Tenant: Separate team/customer Ingester: Temporary warehouse S3 / MinIO: Permanent storage
