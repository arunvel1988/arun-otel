Alloy Processing and Exporting Metrics

Cleaning, modifying and sending metrics to storage systems

Why Processing Metrics is Required?

After Alloy collects metrics from applications, sending them directly is not always a good idea.

In production companies thousands of services generate millions of metrics. Before storing them we need to improve them.

Example raw metric:
cart_count_total 500
Problem:
  • Which application generated this?
  • Which environment?
  • Which team owns it?
  • Is this metric even required?
Alloy processing solves this.

1. Adding Labels

Labels add extra information to metrics. Without labels:

checkout_count_total 200

We only know checkout count. Nothing else.

After Alloy adds labels:
checkout_count_total{ service="checkout", environment="production", team="payments", region="india" } 200
Now we can ask better questions:
Show checkout traffic only for production environment="production" Show payment team's metrics team="payments"

2. Relabeling Metrics

Relabeling means changing existing labels into better labels.

Example: Before:
pod="checkout-app-7sd92"
Problem:

Random Kubernetes pod names are difficult for dashboards.

After relabeling:
service="checkout"

Now dashboards remain stable even when pods change.

Relabel Example in Ecommerce

Before: http_requests_total{ container="cart-v2-6754" } After Alloy: http_requests_total{ service="cart" team="shopping" }

3. Filtering Metrics

Not every metric should be stored. Storage costs money. Too many metrics slow queries.

Example application exposes:
cart_count_total checkout_count_total debug_test_metric temporary_metric
Alloy removes useless metrics:
Stored: cart_count_total checkout_count_total
Benefits:
  • Less storage usage
  • Faster queries
  • Cleaner dashboards

Alloy Processing Example

prometheus.relabel "process" { rule { target_label = "environment" replacement = "production" } rule { source_labels=["container"] target_label="service" } }

This modifies metrics before exporting.

What is Exporting?

After processing, metrics must be sent somewhere. Exporter is responsible for sending metrics to storage.

Exporting to Prometheus

Prometheus is usually used for recent operational metrics.

Examples:
  • Current application health
  • Alerting
  • Debugging today's issue

Alloy
↓
Prometheus
↓

Remote Write Concept

Remote write means pushing metrics into another metrics storage system.

Normal scrape:
Collector pulls metrics
Remote write:
Collector pushes metrics
Flow:
Alloy
↓
Remote Write API
↓
Mimir

Exporting Metrics to Mimir

Mimir is used when metrics become very large.

Example:

A large ecommerce company stores:

  • Millions of requests per minute
  • Thousands of services
  • Years of historical metrics
