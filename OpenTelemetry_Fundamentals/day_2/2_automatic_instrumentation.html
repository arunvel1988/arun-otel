Application Metrics Instrumentation

How applications create metrics using OpenTelemetry

What is Instrumentation?

Your application normally only performs business logic. Example: Customer clicks Add To Cart. The application adds the product into database.

But operations teams need more information.

  • How many cart requests happened?
  • How slow is checkout?
  • How many database calls happened?
  • How many payment failures occurred?

Instrumentation means adding monitoring capability into the application so it can produce metrics.

Application Code → Instrumentation → Metrics → Monitoring System

1. OpenTelemetry Metrics SDK

OpenTelemetry SDK is a library added inside the application. Its responsibility is to create, collect and export telemetry data.

For metrics it provides three important concepts.

Component Purpose
Meter Provider Main object responsible for creating meters
Meter Creates metric instruments like counters and histograms
Exporter Sends metrics outside the application

Ecommerce Flow

Flask Application → OpenTelemetry SDK → OTLP Exporter → Alloy

Creating Metrics using OTel SDK

Example: We want to count every cart click.

Step 1: Create Meter meter = metrics.get_meter("shop-service") Step 2: Create Counter cart_counter = meter.create_counter( "cart_count" ) Step 3: Increase Counter cart_counter.add(1)

Every time customer adds product:

cart_count_total increases 1 2 3 100

Exporter Concept

Creating metrics is not enough. They must leave the application. Exporter sends metrics to another system.

Metric Created → OTLP Exporter → Alloy Receiver → Prometheus / Mimir
Example:
OTLPMetricExporter( endpoint="http://alloy:4317" )

The application does not know about Prometheus or Mimir. It only talks OpenTelemetry.

2. Automatic Metrics Instrumentation

Sometimes developers should not manually create every metric. OpenTelemetry libraries can automatically observe frameworks.

Example: You have 100 APIs. Writing counters manually for every API is difficult. Automatic instrumentation solves this.

HTTP Automatic Metrics

OpenTelemetry watches incoming HTTP requests.

Example:
GET /cart Automatically creates: http.server.duration http.server.request.count
You get:
  • Number of API calls
  • Response time
  • Status codes
  • Error percentage
Request Metric Generated
/checkout checkout latency
/payment failed HTTP 500 count

Runtime Metrics

Runtime means the environment running your code. Example Python, Java or Go runtime.

Automatically collected:
  • Memory usage
  • Thread count
  • Garbage collection
  • CPU usage
Example:
Application Memory: 10 AM = 500 MB 11 AM = 2 GB Possible memory leak

Database Metrics

Database instrumentation observes database communication.

Example: Checkout API:
Checkout → Database Query → Payment
Metrics:
database.query.duration database.connection.count database.errors
Real issue:
Checkout slow Why? Database query taking 5 seconds

gRPC Metrics

Modern microservices communicate using gRPC. OpenTelemetry can monitor these calls.

Example:
Order Service → Payment Service → Shipping Service
Metrics:
grpc.request.count grpc.duration grpc.errors

3. Custom Business Metrics

Automatic metrics understand technology. But they do not understand your business.

For business questions we create custom metrics.

Ecommerce Custom Metrics Examples

Business Question Custom Metric
How many users added products? cart_count_total
How many orders completed? checkout_count_total
How many payments failed? payment_failure_total
How much money earned? revenue_total
