Understanding gRPC

Fast communication between modern microservices

What Problem Does gRPC Solve?

Imagine an ecommerce application. Initially everything is inside one application.

Ecommerce Application
User Cart Payment Inventory Shipping

As the company grows, teams split this into separate services.

User Service ↓ Cart Service ↓ Payment Service ↓ Shipping Service

Now one question appears: How do these services communicate quickly and reliably?

One solution is gRPC.

What is gRPC?

gRPC stands for Google Remote Procedure Call.

It allows one application to call a function running inside another application as if it was a local function.

Normal thinking: Payment service is another application. gRPC thinking: payment.createPayment() looks like a local method call.

Simple Ecommerce Example

Customer clicks checkout.

Checkout Service
calls
Payment Service
Without gRPC:
HTTP POST /payment JSON request JSON response
With gRPC:
paymentService.CreatePayment(order) returns PaymentResponse

Developers work with functions instead of URLs.

Protocol Buffers

gRPC normally does not send JSON. It uses Protocol Buffers.

Protocol Buffer is a contract between services.

Example:
payment.proto service PaymentService { rpc CreatePayment(PaymentRequest) returns (PaymentResponse); } message PaymentRequest { string order_id = 1; double amount = 2; }

Both client and server understand this contract.

Types of gRPC Communication

1. Unary RPC

One request gives one response.

Checkout Service → request payment ← payment success

2. Server Streaming

One request gives continuous responses.

Track Delivery Request once Receive location updates continuously

3. Client Streaming

Client sends many messages. Server replies once.

Upload large file many chunks sent one response received

4. Bidirectional Streaming

Both sides continuously communicate.

Live chat Client sends messages Server sends messages

gRPC and OpenTelemetry

OpenTelemetry can automatically observe gRPC communication.

Example:
Checkout Service ↓ gRPC Call ↓ Payment Service
Metrics generated:
grpc.client.duration grpc.server.duration grpc.requests grpc.errors

These metrics help answer:

  • Which service is slow?
  • Which gRPC call failed?
  • How much latency exists between services?

Simple Memory Rule
