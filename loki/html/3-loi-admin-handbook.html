Loki Administrator Deep Dive

What Does A Loki Admin Own?

A developer thinks:

I want to see my application logs

A Loki administrator thinks:

Who can send logs?
How much can they send?
Where are logs stored?
How long are logs stored?
How expensive is storage?
Is Loki healthy?
Can we survive failures?

1. Tenant Administration

Loki is designed for multiple teams/customers.

Example:
Company Loki Tenant: payments Tenant: orders Tenant: security

How Loki identifies tenant

HTTP Header: X-Scope-OrgID: payments
Flow:
Alloy
adds tenant header
|
Distributor
checks tenant
|
Ingester
stores under tenant
|
S3

2. What Happens If Tenant Sends Too Many Logs?

Example:
Normal: 10 GB/day Bug: DEBUG enabled 2 TB/day
Problems:
  • High storage cost
  • Ingester memory pressure
  • Slow queries
  • S3 cost increase

3. Ingestion Rate Control

Admin protects Loki using limits.

limits_config: ingestion_rate_mb: 20 ingestion_burst_size_mb: 50
Meaning:
Tenant can continuously send: 20 MB/sec

Temporary burst allowed: 50 MB

If limit crossed?

Distributor rejects writes HTTP 429 Alloy retries

4. Per Tenant Limits

Production companies don't give everyone same capacity.

Example:
runtime_config: payments: ingestion_rate_mb:100 development: ingestion_rate_mb:5
Meaning:
Critical service gets more Loki capacity

Development apps are restricted

5. Label Governance

This is one of the biggest Loki admin jobs.

Good labels:
service="payment" environment="prod" cluster="india"
Bad labels:
userid="12345" requestid="abcd1234" transactionid="99999"
Why bad?
Each unique label combination creates a Loki Stream

6. Cardinality Explosion

Example:
100 services OK 100 services + 1 million user ids DISASTER
What happens internally:
Millions of streams | Ingester memory increases | CPU increases | Queries slow | Loki unstable
Admin protection:
limits_config: max_global_streams_per_user: 10000

7. Storage Management

Loki admin owns object storage.

Example:
Ingester | creates chunks | compresses | uploads | S3 / MinIO
Admin monitors:
  • Bucket growth
  • Storage cost
  • Failed uploads
  • Retention cleanup

8. Retention Management

Question:
How long should logs live?
Examples:
Security logs: 180 days Debug logs: 7 days Application logs: 30 days
Config:
limits_config: retention_period:720h

9. Compactor Role

Compactor is the garbage collector.

Flow:
Old TSDB blocks | Compactor | Merge indexes | Delete expired chunks
If compactor dies:
Logs continue. Queries continue. But cleanup stops. Storage grows forever.

10. Can Admin Decide What Goes To S3?

Answer:
Yes. But preferably BEFORE Loki.
Best place:
Application | Alloy | drop useless logs | Loki | S3
Example:
Drop: DEBUG health checks temporary noise

Production Rule

Loki is not a trash bin. Do not store everything. Collect useful logs. Drop noise early. Control tenants. Control labels. Control retention.
