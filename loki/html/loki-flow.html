Grafana queries Loki

Step 1 - Application Creates Logs

Application does not know anything about Loki. It only writes logs.

logger.error("payment failed")

Inside Docker this becomes:

/var/lib/docker/containers/container-id/container.log

Step 2 - Alloy Collects Logs

Alloy is the log collector. Its responsibility:

  • Find log files
  • Read new lines
  • Parse JSON
  • Create labels
  • Send to Loki
Docker File
|
v
loki.source.file
|
v
loki.process
|

loki.write

Step 3 - Loki Receives Logs

Loki receives streams. Example:

service="payment-service" level="ERROR"

Loki separates:

Labels
|
Stored in Index

Actual Log Messages
|
Compressed into Chunks

Step 4 - MinIO Stores Logs

MinIO acts like AWS S3. Loki uploads log chunks here.

MinIO Bucket: loki-data

Objects: chunks/ index/

Step 5 - Searching Logs

Grafana never talks directly to MinIO.

User searches in Grafana
|
Loki receives query
|
Loki checks index
|
Downloads chunks from MinIO
|
Returns logs

LogQL Examples

All payment logs:

{service="payment-service"}

Only errors:

{level="ERROR"}

Failure Scenario 1: Application Down

Application X
|
No new logs generated

Old logs are still searchable from MinIO

Failure Scenario 2: Alloy Down

Application running
|
Docker logs created
|
Alloy X

Result:
Loki receives no new logs

Failure Scenario 3: Loki Down

Application
|
Alloy
|
Loki X

Result:
MinIO gets nothing

Failure Scenario 4: MinIO Down

Application OK
Alloy OK
Loki OK
MinIO X

Result:
Loki cannot flush chunks to storage

Final Understanding

Application = Producer
Alloy = Collector + Processor
Loki = Index + Query Engine
MinIO = Long Term Storage
Grafana = Visualization
